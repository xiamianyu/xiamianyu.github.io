管理成长记

Posted on 2018-07-22   |   In 团队建设   |     |   visitors

管理启蒙

大三的时候有幸加入了 W 学生团队，进入团队的第一个月是学习期，师兄列了一份培训计划，大概内容有 Java 基础、Android 基础、代码规范、项目结构规范、Git 版本控制以及阶段项目实战等。而代码规范、项目结构规范和 Git 版本控制是我之前从来没接触过的，是它们把我从野蛮堆中拯救出来。

W 团队平时的学习或工作是怎么样的呢？以我所在的 Android 部门为例：

  1. 每天列出任务清单，下班日结；
  2. 每两天一次例会，总结前两天，计划明后天，时间不超过 30 min；
  3. 项目开发期间，每天早上站立会，时间不超过 15 min；
  4. 每个项目会指定一个项目主负责人，主负责人把控整体进度和代码规范等；
  5. 不定期举行分享会，不限于技术。

在 W 团队期间，有两个问题一直困扰这我，一是项目结构规范，二是 Git 版本控制策略。项目结构规范，我们遵循的是类似于《Android 权威指南》上的 MVC 架构，不知道外面是怎么样的。Git 版本控制策略基本上没有，开发和发布基本都在 dev 分支上，跟使用 SVN 差不多吧。当时特别渴望出去实习，寻找答案，所以大四下学期我就出来工作了。

参加工作

大四下学期来到广州，经过一个星期的面试，终于在 U 公司安定下来，内心无比的鸡冻，毕竟也是工作一族了嘛。在 U 公司 Android 部门工作了一个月后，我发现它有以下特点：

  1. 团队只有口头开发规范，一个项目有好几种命名规范；
  2. Git 版本控制过于随意，如 commit 信息就写个日期；
  3. 杂碎项目比较多，经常一个人负责几个项目；
  4. 有人经常加班，有人早早下班；
  5. 团队成员之间不知道对方最近在忙什么；
    …

我入职后被安排独立开发一个新闻客户端，由于急着上线，中途安排一个搭档与我一起奋战。让我意想不到的是搭档 1 个月后突然跑路，项目交接匆匆忙忙，草草了事，这时候也没有空闲的人手来帮我，只能是我一个人包揽整个项目。由于搭档的命名规范跟我的相差太多，有点洁癖的我加班加点，硬是把他负责的模块重写了一遍。我暗自发誓，永远不要再做这种吃力不讨好的事了！于是我把之前在 W 团队的规范稍作整理，推荐给组长，组长当时回了一句「参考网上的就可以了」，我内心毫无波动，甚至织起了毛衣。

我想你们可能会说干脆辞职另找一家算了，刚开始我也是这么想的，但后面就打消了这个念头，我只是被拒绝了一次，一次就气馁了？这不像我喔。

接下来的一段时间，我在网上查阅了大量的开发规范资料，吸取精华的部分，整理出一份比较齐全的文档，然后发到团队讨论组上，大家也只是口头敷衍一下，又继续蛮干了。

带新人

有一次 CTO 找到团队成员 A 问，怎么对接个接口要这么久？

A：这个接口字段太多了，一个个 JSON 字段取出来要花些时间。
CTO（苦笑）： Android 没有 JSON 解析工具吗？

我当时坐在旁边就顺便回了一句：有，GSON。谁能想到就这一句话竟然是我职业生涯的转折点。

几个月后，CTO 突然叫我去谈话，大概的意思是目前 Android 团队工作效率很低，看出我比较有潜力，想找个新人给我带，看能否把工作效率提上来。当时我才入职 9 个月，有点受宠若惊，也怕不能胜任，后面想想怎么也得试试吧，机不可失。

晚上回去就睡不着了，心想怎么带新人呢？这时我想起来在 W 团队的经历，师兄怎么带的我，我就怎么带新人不就得了吗？

第二天我跟新人做了个全方位的了解，根据他的情况指定一份 1 个月后能实战的培训计划。培训计划跟在 W 团队时的差不多，另外针对他薄弱的部分加以强化以及补充 UML 的学习。

培训完毕后，刚好有一个为期 1 month 的小项目，我跟组长争取让新人来开发，我辅助（不参与开发）。新人的成长速度比我想象的要快，项目提前开发完成，顺利收工。我这才长舒一口气，带人这事总算不负所托。

带团队

16 年 5 月份，组长突然离职并推荐我上位，事情来得有点突然，没有一点心理准备。组长离职后，团队就剩下 3 个人，其余两人入职都还不到 1 年，入戏不深，我觉得这是团队洗心革面的好机会。

在带新人的过程中，我同时在思考一个问题：如果我当组长，该怎么管理团队，一个团队需要具备什么马车？很幸运的是我在老司机的窝里看到一本「App 研发录」，翻开目录的那一刹那，我欣喜若狂，这不就是我迫切需要的吗？看完「App 研发录」后，我立刻着手拟定一份旨在提高团队工作效率的团队建设规划图，原稿如下：

除此之外，团队开发规范、项目结构规范和 Git 版本控制策略之前都已经准备完毕，所以我上位后要做的就是执行这些计划。

计划落地

所谓 5% 在策略，95% 在执行。计划实施过程异常艰辛，活生生得把我变成一个霸道哥。第一个要落地的计划是统一开发规范，因为它是基石。

开发规范，据我所知，很多公司都是比较混乱的，统一开发规范，如何落地？如果你把它看得跟国家统一一样重要，那事件就简单许多了，因为它上升为原则问题，原则没得商量，你要么遵守，要么出门右拐。那是不是一两天就能彻底落地好呢？不是，适应新的开发规范需要改变一个人的习惯，我给组员的时间是 3 个月，三个月后必须基本纠正。除了团队技术博客夭折之外，其余计划半年后基本完成。

跨部门协作

当把团队内部梳理得七七八八的时候，一个更大的问题来了，它就是跨部门协作。一个 App 从需求分析到部署上线，会有产品组、设计组、测试组和开发组等参与，每个阶段各组都会产生相应的文档。而我们当时习惯性地用 QQ 讨论组传文件，每次都是全量下载，很多时候会弄错版本造成信息不同步。后来便和其他组的组长约定好一套文档管理规范，用 SVN 统一管理起来，形成了项目文档。这样开发期间不仅方便跨组协作，同时有利于项目后期的维护和交接。当然文档统一管理只是跨部门协作中冰山一角。

管理者的自我修养

前面零零碎碎讲了一些团队建设方面的内容，下面我们来聊聊管理者的自我修养。

角色转变

担任组长的 1 个月里，我还将自己当成组员，跟组员「抢」工作，而自己已经开始参与中层事务的管理，一天下来就没多少开发时间，只能加班赶工。我就在想，组长的职责是什么？很幸运我在「格鲁夫给经理人的第一课」中找到了答案，组长的职责应该是为组员搭建一个高效的工作平台、合理安排任务、协调资源、把控风险、技术引导等，因此我就辞去大部分开发任务。

树立威严

威严这块，我吃过一次亏。大四上学期我在一所中学教育实习，当时领导对我说，第一次和学生见面一定要树立威严，不能随便，否则你后面很难管。我不以为然，我行我素，果然没几天我就中招了。其实管理者何尝不像老师呢？

建立威信

建立威信的方法是持续做正确的事。

用好试用期

我身边有些管理者会反馈一个问题：招的人转正后没怎么听话。我问他们试用期都对新人干了什么？他们说就做项目呗，这就是问题所在了。试用期是新人转正的过渡期，这期间主要是考核新人的适应能力、技术能力、沟通表达能力、抗压能力、责任心和进取心等，如果这期间你对新人放任自如，后面烂摊子有你收拾的。

用人

有时候觉着自己不像个管理者，更像个老师。因为我希望自己的组员都能有出息，这让我鸭梨山大，还好后来想通了：我不是老师。团队中有人技术强，有人责任心强、有人协调能力强，有人就打打杂混日子。重要的任务当然是交给能力强的人主导，一般般的、杂碎的任务就交给打杂的处理。也许你们会埋怨管理者太偏心，那为啥不学学隔壁老王呢？

总结

以上就是我快 2 年的管理之路以及感悟，我说的不一定对。
