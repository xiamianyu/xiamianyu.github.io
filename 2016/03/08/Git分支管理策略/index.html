Git分支管理策略

  • Git
  • 分支管理

Git的分支就像一把双刃剑，用对了风生水起，反之引来杀身之祸，哈哈，说得有些严重了。不管是团队开发还是个人开发，也不管是使用Git或是SVN，良好的版本管理策略定能使你事半功倍。

这里有一篇来自国外牛人Vincent Driesse分享的「A successful Git branching model」，对应的中文翻译在这里，写得很不错，受益匪浅。

下面是我对该文章的总结，作为学习的笔记吧。

分支管理模型

主分支

主分支是所有开发的核心分支。所有开发的产出（线上软件）最终都会反映到主分支的代码中。主分支由 master 和 develop 分支组成且且永久存在。

master 分支

存放随时可供在生产环境中部署的代码（Production Ready state）。当开发活动告一段落，产生了一份新的可供部署的代码时，master分支上的代码会被更新。同时，每一次更新，最好添加对应的版本号标签（TAG）。

develop 分支

当 develop 分支上的代码已实现了软件需求说明书中所有的功能，通过了所有的测试后，并且代码已经足够稳定时，就可以将所有的开发成果合并回 master 分支了。

辅助分支

辅助分支是用于解决特定问题的分支。主要用于软件新功能的并行开发、简化新功能开发代码的跟踪、辅助完成版本发布工作以及对生产代码的缺陷进行紧急修复工作。辅助分支与主分支不同，通常只会在有限的时间范围内存在。

辅助分支包括：

  • 用于开发新功能时所使用的 feature 分支；
  • 用于辅助版本发布的 release 分支；
  • 用于修正线上产品的代码的缺陷的 hotfix 分支。

feature 分支

使用规范：

  • 可以从 develop 分支派生
  • 必须合并回 develop 分支
  • 命名 feature-*

feature 分支（有时也叫 topic 分支）通常是在开发一项新功能的时候使用，这个分支上的代码变更最终合并回develop分支或者干脆被抛弃掉。

一般而言，feature 分支保存在开发者本地仓库中而不提交到远程仓库中。

release 分支

使用规范：

  • 可以从 develop 分支派生
  • 必须合并回 develop 和 master 分支
  • 命名 release-*

release分支是为发布新的产品版本而设计的。在这个分支上的代码允许做小的缺陷修正、准备发布版本所需的各项说明信息（版本号、发布时间、编译时间等等）。通过在release分支上进行这些工作可以让develop分支空闲出来以接受新的feature分支上的代码提交，进入新的软件开发迭代周期。

当develop分支上的代码已经包含了所有即将发布的版本中所计划包含的软件功能，并且已通过所有测试时，我们就可以考虑准备创建release分支了。而所有在当前即将发布的版本之外的业务需求一定要确保不能混到release分支之内（避免由此引入一些不可控的系统缺陷）。

hotfix 分支

  • 可以从 master 分支派生
  • 必须合并回 develop 和 master 分支
  • 命名 hotfix-*

当线上的软件遇到了异常情况或者发现了严重到必须立即修复的软件缺陷的时候，就需要从master分支上指定的TAG版本中派生hotfix分支来组织代码的紧急修复工作。

这样做的显而易见的好处是不会打断正在进行的develop分支的开发工作，能够让团队中负责新功能开发的人与负责代码紧急修复的人并行的开展工作。
